# Phase 4.5 — Smoke Tests Design

## Goal

Regression test suite covering core functionality before Phase 5 branding changes. Provides a safety net so restyling doesn't silently break user flows.

## Prerequisites

### Supabase test project
- Create `plexease-test` project under the existing Supabase org
- Apply schema from `schema.sql` (source of truth)
- Create the `increment_usage` RPC function
- Populate `.env.test` with test project URL, anon key, and service role key

### Playwright installation
- Install `@playwright/test` as a dev dependency
- Install browser binaries (`npx playwright install`)

## Global Setup & Teardown

### `global-setup.ts`

1. Create Supabase admin client using service role key
2. For each test user (`free@test.plexease.io`, `pro@test.plexease.io`):
   - `auth.admin.createUser({ email, password, email_confirm: true })`
   - If user already exists (409), continue (idempotent)
3. For the pro user: upsert a `subscriptions` row with `plan: 'pro', status: 'active'`
4. Launch browser, log in as each user, save `storageState` JSON files

### `global-teardown.ts`

1. Reset usage counts for both test users
2. Reset pro user's subscription to known good state
3. Clean up any test-created data that could leak between runs

Users persist between runs (no create/delete churn). Only mutable state is reset.

## Custom Fixtures (`fixtures/index.ts`)

Extends Playwright's `test` object:

- `freeUserPage` — `Page` pre-loaded with free user's `storageState`
- `proUserPage` — `Page` pre-loaded with pro user's `storageState`
- `anonPage` — plain `Page` with no auth state
- `supabaseAdmin` — Supabase client with service role key for DB seeding

## Page Object Model

Simple classes with locators and action methods. No base class hierarchy, no inheritance.

Example:
```ts
export class LoginPage {
  constructor(private page: Page) {}

  readonly emailInput = this.page.locator('input[name="email"]');
  readonly passwordInput = this.page.locator('input[name="password"]');
  readonly submitButton = this.page.locator('button[type="submit"]');

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}
```

## Stripe Checkout Strategy

The checkout test uses a two-tier assertion approach:

1. **Hard assertion:** Clicking checkout redirects to `checkout.stripe.com` — this must pass
2. **Soft assertion:** Wrapped in try/catch — attempt to fill test card (4242 4242 4242 4242), complete payment, verify success page. If Stripe's hosted checkout UI changes selectors, the test passes with a console warning rather than failing the suite

This gives regression signal on the integration without brittleness from third-party UI.

## Usage Limit Strategy

DB seed via Supabase admin API (`setUsageCount`) to 19, then:
1. Real API call to NuGet Advisor — succeeds (usage now 20)
2. Another real API call — blocked with upgrade prompt

Tests the real enforcement logic with minimal API calls (2 Claude calls instead of 20).
